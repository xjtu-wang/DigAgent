# 审批通路大改：action digest 不再与挂起动作匹配

## Context

会话 `sess_13483fc8f977`（任务：分析 www.xjtu.edu.cn）在 `node_subagent_web_analysis` 阶段以 `block_reason: "approved action digest no longer matches the pending action"` 整个 run `failed`。这是当前审批通路的根因型脆弱性，并非边缘 bug。已知的几次 run 失败（包括之前的 xjtu 场景）都卡在"审批"与"动作重建"的不一致上。需要一次较大范围的重构，把审批通路的状态机、摘要算法与子代理子动作的承载方式理清。

相关代码与现象：

- `src/digagent/agent/runtime.py:941`（`approve()`）与 `:2354`（`_authorize_action` 复用已 APPROVED 审批时的二次校验）使用同一条严格比较：`approval.action_digest != self._approval_digest(current_action)` → 不匹配即 `_fail_run`，没有任何降级路径。
- `_approval_digest`（`:3139`）输入为 `{run_id, actor_agent_id, name, arguments, targets, node_id}`，靠 `action_digest` 做 canonical JSON。任何 `arguments` 键序不同、targets 列表顺序变化、LLM 重新生成时新增/删除可选键都会改变摘要。
- `_authorize_action`（`:2263`）无条件把 `node.action_request = action.model_dump(...)`；对 subagent 节点，内部 worker tool 动作会直接覆盖该节点原先的 subagent 动作。
- `_approval_action_for_node`（`:987`）取 `worker_pending_action` → `node.action_request` → 重建。subagent 场景下二者都可能在被审批后被下一轮 worker 提议刷新。
- `_execute_worker_tool_action`（`:1960`）每次 worker 产出新动作就会：1) 写 `node.metadata["worker_pending_action"]` 覆盖上一次；2) 再次调用 `_authorize_action` 覆盖 `node.action_request`。复用同一个 `approval_id`（`_get_or_create_approval`，`:2365`）时，审批记录里的 `action_digest` 保持不变，但节点当前动作已被刷新——摘要自然对不上。
- `_fail_run`（`:3375`）做终结态、清 pending，没有"重新发起新的审批"回路；主循环（`:817` 周围）也不会因 digest 失配而 replan。
- 前端 `resolveApproval` 一旦提交成功即 `loadSessions` 并依赖服务端事件更新——服务端 `_fail_run` 之后 UI 只会看到 run 失败信息，不会看到"动作已变、请重新审批"的恰当引导。

目标：消除"正常路径上的审批-动作漂移 → 整个 run 死亡"这条通路；把子代理子动作的承载从"复用 subagent 节点"改为独立状态；把摘要改为 stable / policy-equivalent；digest 失配降级为"重发审批 / 等待新审批"而非失败。

## Recommended Approach

### 1. Audit：先把当前行为锁在回归测试里

在 `tests/` 下新增 `test_approval_digest.py`（若不存在），覆盖三条关键路径，随后按本计划迁移再让它们通过：

- subagent 节点第一次请求 → 发起审批 → 同一动作被再次提议 → 期望不再 fail_run。
- 审批挂起期间 worker 重新生成"等价但键序不同"的 arguments → 期望摘要仍匹配。
- 主动重规划（replan）后节点被废弃（`SUPERSEDED`）→ 老 approval 自动失效而不是触发 fail_run。

### 2. 稳定的摘要输入（`src/digagent/agent/runtime.py::_approval_digest` 与 `digagent/utils.py::action_digest`）

- 在 `_approval_digest` 里对 `arguments` / `targets` 做规范化：
  - 递归字典按 key 排序（`action_digest` 已 `sort_keys=True`，但嵌套 list 不会被排序；`targets.paths`、`targets.domains` 强制排序+`normalize_domain`/`Path.resolve`）。
  - 剔除纯"展示性"键（`justification`、`expected_artifacts`、`created_at`、`action_id`）——当前虽未进摘要但值得在 schema 层固化为"审批不变量"集合。
  - 对白名单工具可选择性地 canonicalize 参数（例如 `web_fetch` 的 `url` 去末尾斜杠、转小写主机名）。新增 `src/digagent/agent/approval.py` 收敛这部分逻辑；不要把业务 canonicalization 塞进工用 `action_digest`。
- 加 `approval.policy_key` 字段（schema 可选），用"动作类型+规范化 targets+规范化关键 arguments"生成，仅用于"是否实质等价"判断；`action_digest` 保留为严格指纹用于审计。失配时只有当 `policy_key` 也不等才视为不等价。

### 3. Subagent 节点与子动作解耦（`src/digagent/agent/runtime.py`）

核心 invariant：**subagent 节点的 `action_request` 永远是它自己的 delegation 动作**，不因 worker 的子动作而改动。

- `_authorize_action` 新增参数 `persist_on_node: bool = True`，子动作走 `persist_on_node=False`，不写 `node.action_request`（但仍可写 `node.metadata["worker_pending_action"]`）。
- 由 `_execute_worker_tool_action` 明确传 `persist_on_node=False`；主 tool/skill/export 路径保持 `True`。
- 为每个 worker 子动作单独建"子审批"（`ApprovalRecord.parent_approval_id` / `ApprovalRecord.kind = "worker_tool"`），其 `action_digest` 基于子动作本身；不复用节点主审批。
- `_get_or_create_approval` 查询 key 从"node.approval_id 单字段"改为"(node_id, action_policy_key)" map：同一策略的子动作共享审批；不同策略的新动作拿到新的 approval，旧审批 `SUPERSEDED` 并 emit 事件。

### 4. digest 失配的降级路径（`approve()` 与 `_authorize_action`）

- 现有两处 `_fail_run("approved action digest no longer matches …")` 全部替换为：
  1. 标记原 approval `status = SUPERSEDED`，emit `approval_superseded`；
  2. 为新动作调用 `_get_or_create_approval`（拿到的将是新 approval，因为 policy_key 变了）；
  3. 若用户曾给 auto_approve 或之前是同 policy_key 的 APPROVED，则沿用决定，无需再次打扰；
  4. 否则把 run 重新置为 `AWAITING_APPROVAL` 并 emit `approval_required`，不 fail_run。
- 不保留严格模式兜底（用户选型）：digest 失配统一走 supersede，不再 fail_run。`_fail_run` 仅用于真正的策略 DENY/budget 超限/明确拒绝。

### 5. 事件/前端（`src/digagent/api.py`、`webui/src/use-workspace-controller.js`、`chat-timeline.jsx`）

- 新增事件：`approval_superseded`（携带 old_approval_id、new_approval_id、policy_key、diff 摘要）。
- 前端策略（用户选型：旧卡片变灰 + 新卡片独立出现）：
  - `use-workspace-controller.js` 的 `resolvedApprovalIds` 扩展为 `inactiveApprovalIds`，把 `approval_superseded.old_approval_id` 一并并入；
  - `ApprovalCard` 在 inactive 下渲染灰态 + "已被新动作替代 → apr_xxx" 跳转链；
  - 新 approval 通过常规 `approval_required` 事件以独立卡片出现在时间线末尾，可正常点击批准/拒绝；
  - 按钮在 inflight / inactive 下均 disabled。
- 文案：区分"你批准的动作已被新动作覆盖（需再次确认）"与"审批被拒"两类；`approval_superseded` 在时间线中以系统提示条呈现，不占用 ApprovalCard 卡位。

### 6. 关键文件

- `src/digagent/agent/runtime.py`：`_approval_digest`、`_authorize_action`（新增 `persist_on_node`）、`approve`、`_execute_worker_tool_action`、`_get_or_create_approval`、`_approval_action_for_node`，外加新建 helper（policy_key / 子审批）。
- `src/digagent/utils.py`：`action_digest` 保持；补一个 `stable_canonical(payload)` 专用于摘要前的规范化。
- `src/digagent/agent/approval.py`（新建）：集中 digest / policy_key / canonical 规则。
- `src/digagent/models.py`：`ApprovalRecord` 增补 `policy_key`、`parent_approval_id`、`kind`、`superseded_by` 字段；`ApprovalStatus` 增 `SUPERSEDED`。
- `src/digagent/api.py`：在 approve 响应里返回 `superseded_by` 与新 approval 摘要。
- `webui/src/use-workspace-controller.js`：派生 resolved/superseded 集合；`chat-timeline.jsx::ApprovalCard` 渲染三态。

### 7. 可复用的已有能力

- `utils.action_digest`（`utils.py:36`）的 canonical-JSON + sha256 基础不变。
- `_get_or_create_approval`（`runtime.py:2365`）作为审批池入口保留，只扩索引 key。
- `_approval_challenge_value`（`runtime.py:3301`）仍作为 challenge token 基础。
- `approval_resolved`/`approval_required` 事件流保留，新增事件跟既有模式一致。

## Verification

1. **回归测试**：新增 `tests/test_approval_digest.py`：
   - (a) subagent 首次 delegate → 内部 worker `web_fetch` 请求被批准 → worker 再次 regenerate 同 policy_key 动作不触发 fail_run；
   - (b) worker 改用不同 URL → 发起 `approval_superseded` + 新 approval；
   - (c) 手工篡改 approval record 的 `policy_key` 使其与节点当前动作不等价 → approve() 不 fail_run，改为发送新审批。
2. **端到端回放**：用 `sess_13483fc8f977` 的 user_task 重跑（`digagent run --task "分析一下www.xjtu.edu.cn" --domain xjtu.edu.cn`），期望：
   - subagent 的 web_fetch 请求正常被审批/自动批准；
   - run 走完到 report 生成；
   - audit ndjson 中无 `"digest no longer matches"`，但可以出现 `approval_superseded`。
3. **摘要稳定性单测**：给 `_approval_digest` / `policy_key` 写若干"等价变形"输入（参数键序、targets 列表顺序、URL 尾斜杠、大小写域名），断言摘要/policy_key 一致。
4. **前端手工验证**：在 webui 里复现"批准后被新动作取代"场景，确认卡片文案从"需要审批"→"审批已处理（已被新动作取代）"→新卡片出现并可批准；按钮在 inflight / superseded 下均 disabled。
5. **审计完整性**：审批链（老 approval、approval_superseded 事件、新 approval、最终 APPROVED）在 audit ndjson 中形成可追溯序列，不出现孤儿 pending_actions。

## Migration / 风险

- `ApprovalRecord` 新增字段使用 Optional 默认值，旧 session 文件无需迁移，加载时兜底。
- 引入 SUPERSEDED 状态后，定时 `_expire_pending_approvals` 类逻辑要能识别非活跃状态不再过期通知用户。
- 子审批的引入会影响 `run.approval_ids` 列表与 `session.pending_approval_ids` 的"谁负责清理"——需在 `_execute_worker_tool_action` 成功/失败路径上显式移除子审批。
- 行为变更：之前 digest 不匹配直接挂掉，现在统一改为弹新审批（用户确认"一刀切"），不保留严格模式开关。需在 release note 中明确这是语义变化。
