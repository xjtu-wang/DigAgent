# agent 的实现目标
可以 CLI 和 WebUI 启动的 agent，针对安全渗透分析领域，面向常见 CTF 题、源码分析、Web 渗透分析，出具 writeup 或渗透分析报告，支持导出 Markdown 或 PDF；可以载入 skill 市场的 skill，例如 [CTF-SKILL](https://github.com/GALIAIS/CTF-Sandbox-Orchestrator)；权限管控针对工具调用设计，针对不同的 agent 设计不同的权限管控策略；多 agent 及 subagent（supervisor pattern）设计，可以加载不同的 prompt、permission 配置，memory 由主 agent 持有，subagent 只做专项任务；从漏洞知识库提取知识，一键下载转换漏洞知识库。

# agent的工作形式
不以单个任务为分界，而以session为最小粒度单元，用户可在一次对话中给出多个命令，Sisyphus可以回答用户关于渗透过程中的问题

## Session 工作流与状态模型

### 用户消息处理分派
每条用户消息经 `SessionRouter` 后落入以下四种结果之一：

- **direct_answer**：用户询问进度、证据、审批理由等过程性问题。系统直接基于当前上下文回答，不创建新 run。
- **create_run**：session 处于空闲状态，用户提出新任务。初始化新 run 进入执行图。
- **continue_run**：当前 run 处于等待审批或等待用户输入，用户提供审批结果或补充信息。继续执行该 run。
- **reject**：存在未完结 run，但用户发起新的独立任务。提示等待当前 run 完成或取消。

### Session 状态

- **idle**：无活跃 run，可接受新任务。
- **active_run**：存在正在执行（非等待用户）的 run。
- **awaiting_approval**：run 暂停等待人工审批。
- **awaiting_user_input**：run 暂停等待用户补充信息。
- **archived**：会话已归档，可继续聊天。

### 关键交互规则

- **过程性问答**：询问进度、已知证据、下一步计划、审批原因时，直接回答，禁止创建 run 或调用工具。
- **多命令同条消息**：先拆分意图，顺序串行执行。首个意图创建 run，后续相关意图作为 task graph 的补充上下文或后续节点。不允许多 run 并发。
- **继续/中断/改计划**：继续、批准、补充信息作用于当前 run。修改范围可在计划阶段修订。取消任务将 run 置为 cancelled。

### Session 与 Run 关系

- **生命周期**：Session 是长期对话容器，Run 是单次受控任务。
- **数据持有**：Session 持有消息列表、范围、run 列表；Run 持有计划、证据、审批记录、产物。
- **状态控制**：Session 状态决定能否创建新 run，同一时刻最多一个 active_run；Run 内部可并发 subagent。
- **写入权限**：Session 只追加消息和 run 引用；Run 负责写入证据、产物、报告。


# 技术基础
使用 [deepagents](https://docs.langchain.com/oss/python/deepagents/overview)、[langchain](https://docs.langchain.com/oss/python/langchain/overview)、[langgraph](https://docs.langchain.com/oss/python/langgraph/overview) 生态进行 agent 构建；使用 React + Vite + 组件库进行 WebUI 构建。

# MVP 边界
## 范围
- 当前 MVP 范围以本文末尾的“验收标准”为准，不额外扩展更复杂的平台能力。
- 当前里程碑即“功能验收 / 集成验收 / 安全验收”三组标准。
- 设计目标优先保障“可跑通、可审计、可导出、可受控”，不优先追求高度自治或复杂自治策略。

## 非目标
- 非目标不是功能缺失，而是主动避免过于复杂的设计。
- 本阶段不做分布式 agent 调度。
- 本阶段不做多租户账号体系与复杂 RBAC。
- 本阶段不做向量数据库、RAG 编排平台化，只保留可替换接口。
- 本阶段不做完整 skill 市场平台，只支持本地加载和兼容外部 skill 包格式。
- 本阶段不做自动漏洞利用链的大规模自治执行，只做受控分析和有限工具调用。

# 主流程
用户输入 -> 意图识别 -> 主 agent 制定计划 -> 权限预判 -> multi-agents 分工协作 -> 汇总证据 -> 生成报告或 writeup -> 导出 Markdown/PDF

# 运行时图设计
## 设计原则
- `run` 的唯一执行真相是任务 DAG，不再维护与之并行的固定运行节点图。
- 图由 Sisyphus 作为 root supervisor 持有，root agent 负责规划、局部重规划、证据汇总、memory 提交和报告生成。
- subagent 只接收任务切片、临时上下文和受限权限，不直接写长期 memory，不直接结束整个会话。
- 图结构允许分支与汇合，但每个节点都必须有明确依赖、状态和恢复点。
- `task_type` 只作为兼容展示提示，不能主导规划、报告或记忆；真正的语义真相是 `intent_profile + task_graph + evidence`。

## 运行状态
一次完整运行记为一个 `run`，状态机如下：

```text
created
  -> planning
  -> awaiting_approval
  -> awaiting_user_input
  -> running
  -> aggregating
  -> reporting
  -> completed

任意阶段都可进入:
failed
cancelled
timed_out
```

状态含义：
- `created`：收到用户任务，初始化 run 上下文。
- `planning`：拆解目标、识别风险、生成初始 task graph。
- `awaiting_approval`：某个 task node 需要人工审批，图暂停在该节点。
- `awaiting_user_input`：某个 task node 需要用户补充输入，图暂停在该节点。
- `running`：调度 ready 节点，执行 tool、skill、subagent。
- `aggregating`：执行聚合类节点，整理证据与阶段结论。
- `reporting`：执行报告与导出节点。
- `completed`：所有必要节点完成，且最终输出已生成。
- `failed`：出现不可恢复错误，或图无法继续推进。
- `cancelled`：用户主动取消。
- `timed_out`：达到运行超时或审批超时。

## Task Graph
后端真实运行结构是一个 DAG：

- `TaskNode`
  - `node_id`
  - `title`
  - `kind`
  - `status`
  - `description`
  - `depends_on`
  - `children`
  - `summary`
  - `block_reason`
  - `action_request`
  - `approval_id`
  - `evidence_refs`
  - `artifact_refs`
  - `retry_count`
  - `max_retries`
  - `metadata`
- `TaskEdge`
  - `source`
  - `target`
- `TaskNodeStatus`
  - `pending`
  - `ready`
  - `running`
  - `waiting_approval`
  - `waiting_user_input`
  - `completed`
  - `failed`
  - `blocked`
  - `deprecated`

## GraphManager 与调度
- `GraphManager` 负责维护 DAG、校验无环、计算 ready 集、更新 completed/blocked/deprecated 集、应用 graph edit operations。
- 初始任务图采用 `LLM-first` 规划，由 planner agent 优先产出结构化 DAG；只有在离线模式、模型不可用或图解析失败时，才退回 deterministic fallback graph。
- Scheduler 按依赖拓扑选择 ready 节点推进，而不是扫描线性步骤列表；有限并行由 `max_parallel_tools` 与 `max_parallel_subagents` 控制。
- 节点完成后自动尝试解锁下游节点；若图进入无 ready、无 active、且未满足结束条件的状态，则进入局部重规划或失败。
- 局部重规划通过 `GraphEditOp` 完成，最少支持：
  - `ADD_NODE`
  - `UPDATE_NODE`
  - `ADD_EDGE`
  - `REMOVE_EDGE`
  - `DEPRECATE_NODE`
- 当 scope 不足时，不直接 reject 用户，而是创建 `input` 类型 clarify 节点，由 agent 生成一条简短自然的追问并进入 `waiting_user_input`。
- clarify 节点的用户可见问题统一存放在 `TaskNode.metadata.question`；进入等待态时，后端必须把这条问题同步追加为一条 `assistant_message` 到聊天流。
- clarify 等待阶段不允许同时存在可执行业务节点；若用户补充信息后触发重规划，新 continuation 必须替换 clarify 之后的旧 speculative 分支，旧分支节点统一标记为 `deprecated/superseded`，不得再进入 ready 集。

## Agent 分工矩阵
- `sisyphus-default`：root supervisor，负责 session 路由、审批协调、最终对话输出和整体 run 生命周期控制。
- `prometheus-planner`：负责初始 DAG 规划、clarify 追问、用户补充后的重规划，以及 ready node 批次选择。
- `hephaestus-deepworker`：负责通用 repo / web 深挖类 subagent 任务。
- `hackey-ctf`：负责 CTF 专项分析和 skill 驱动的题目求解。
- `report-writer`：负责基于 `intent_profile + task_graph + evidence` 生成最终报告草稿。
- `memory-curator`：负责 daily note、长期记忆提升和 wiki 更新。
- 除用户显式指定 root profile 外，specialist / planner / writer / curator 的调度默认由 `sisyphus-default` 自动决定。

## 工具注册
- 工具采用 `manifest + adapter` 方式注册。
- `data/tools/*.yaml|json` 负责声明工具名字、描述、风险标签、默认目标和执行 adapter。
- `GET /api/catalog` 的工具列表直接由 manifest 派生，而不是写死在运行时代码中。
- permission layer 对“已注册动作”的判断以 manifest 为准；未注册工具一律 deny。

## subagent 设计
- subagent 必须由 root agent 显式创建，不能由 skill 直接创建。
- subagent 输入必须是结构化任务，而不是整个 session 原文。
- subagent 只拥有：
  - 当前 task node 的目标
  - 必要 evidence 摘要
  - 限定后的工具和路径权限
  - 临时 scratchpad
- subagent 输出必须回包为结构化结果：

```json
{
  "subagent_id": "hephaestus_01",
  "status": "completed",
  "summary": "找到可疑反序列化入口",
  "evidence_ids": ["ev_x1", "ev_x2"],
  "artifact_ids": ["art_x1"],
  "recommended_next_actions": [],
  "memory_candidates": []
}
```

## 并发规则
- 默认只允许 root agent 有限并发调度节点。
- 工具并发由 `max_parallel_tools` 控制，subagent 并发由 `max_parallel_subagents` 控制。
- 写文件、导出、skill 执行、高风险网络动作视为互斥节点，不与其他互斥节点并发。
- 工具并发与 subagent 并发分开计数，防止单次运行失控。

## 结束条件
满足任一条件即可结束：
- 所有必要 task node 已完成或明确降级，且已有足够证据支撑最终输出。
- 用户主动结束。
- 达到时间、并发、工具调用预算上限。
- 关键步骤被 deny 且不存在可行降级路径。

## 失败与恢复
- tool 执行失败时，允许有限重试，重试次数由 tool config 控制。
- subagent 失败不直接导致整个 run 失败，root agent 先决定是否降级、替换或终止。
- 节点进入 `waiting_approval` 后必须可恢复，审批后从原 `node_id + action_id` 继续，而不是整图重跑。
- 节点进入 `waiting_user_input` 后，用户补充信息后恢复该节点或触发局部重规划。

# 权限模型
权限粒度：tool / skill / filesystem path / network

采用白名单模式，不同 agent 有不同权限：
- `allow`：直接执行。
- `confirm`：需要人工审批后执行。
- `deny`：直接拒绝。

高风险动作默认落入 `confirm`：
- 写文件
- 执行 shell 命令
- 发起网络请求
- 调用外部 exploit/扫描工具
- 导出包含敏感内容的报告

# 权限执行时序
## 核心原则
- 所有动作先形成结构化 `ActionRequest`，再进入 permission layer。
- tool、skill、subagent 都不能绕过 permission layer 直接执行系统能力。
- 审批结果必须绑定到“精确动作摘要”，防止用户批准 A 后被替换成 B。

## ActionRequest 模型
```json
{
  "action_id": "act_xxx",
  "run_id": "run_xxx",
  "actor_agent_id": "sisyphus",
  "action_type": "tool|skill|subagent|export",
  "name": "bash|web_fetch|ctf_skill|pdf_export",
  "arguments": {},
  "targets": {
    "paths": ["/workspace/project/app.py"],
    "domains": ["example.com"]
  },
  "justification": "需要读取源码确认漏洞入口",
  "risk_tags": ["filesystem_read"],
  "expected_artifacts": [],
  "created_at": "2026-04-08T10:00:00Z"
}
```

## 执行时序
```text
Agent Runtime
  -> 生成 ActionRequest
  -> Permission Engine 规范化参数与目标
  -> Policy Resolver 依次检查:
       1. 全局 deny 规则
       2. agent profile allowlist
       3. session scope 限定
       4. 风险分级策略
       5. 速率/预算限制
  -> 得到 allow / confirm / deny

如果是 allow:
  -> Runner 执行
  -> 产出 artifact/evidence
  -> Audit Log 记账
  -> 返回结果给 graph

如果是 confirm:
  -> 生成 approval record
  -> CLI/WebUI 展示审批卡片
  -> 用户批准后生成 approval token
  -> Runner 校验 token 后执行
  -> Audit Log 记账

如果是 deny:
  -> 记录拒绝原因
  -> 返回 policy error
  -> root agent 选择改写计划或结束
```

## 审批对象
审批记录至少包含：

```json
{
  "approval_id": "apr_xxx",
  "action_id": "act_xxx",
  "status": "pending|approved|rejected|expired",
  "action_digest": "sha256:...",
  "requested_by": "sisyphus",
  "requested_at": "2026-04-08T10:00:00Z",
  "resolved_at": null,
  "resolver": null,
  "reason": null
}
```

说明：
- `action_digest` 用于绑定审批和动作内容。
- 只要 action 参数、目标路径、目标域名发生变化，就必须重新审批。
- 审批超时后状态转为 `expired`，图可选择终止或重新发起。

## 执行器约束
- Runner 是唯一可触达系统能力的位置。
- Skill 脚本不直接拿到 shell/network/filesystem 句柄，只能通过受控 tool API 请求能力。
- Runner 在真正执行前再次校验：
  - action_id 是否存在
  - policy decision 是否仍然有效
  - approval token 是否匹配 digest
  - 目标路径、域名是否越界

## 审计日志
每次动作都必须写审计日志，建议使用 NDJSON，便于追加和回放：

```json
{
  "event_id": "audit_xxx",
  "timestamp": "2026-04-08T10:00:01Z",
  "run_id": "run_xxx",
  "action_id": "act_xxx",
  "actor_agent_id": "sisyphus",
  "decision": "allow|confirm|deny",
  "executor": "tool_runner",
  "result": "success|error|blocked",
  "exit_code": 0,
  "artifact_ids": ["art_x1"],
  "evidence_ids": ["ev_x1"]
}
```

# agent 类型
Sisyphus 主指挥官。负责制定计划、分配任务给专家团队，并以激进但受控的并行策略推动任务直至完成。他持有 root session、memory commit、最终报告输出权限。

Hephaestus 自主深度工作者。接收明确目标后独立探索代码库模式或 Web 内容，从头到尾执行专项任务，但只能在受限权限范围内工作。

Prometheus 战略规划师。通过访谈模式，在进行具体任务之前，先通过提问确定范围并构建详尽的执行计划，适合高不确定性任务。

hackey 渗透专家，精通使用 [CTF-SKILL](https://github.com/GALIAIS/CTF-Sandbox-Orchestrator)，可以向其他 agent 提供渗透测试上的指导，但其工具能力同样受 permission layer 约束。

# agent 配置
可配置 `model`、`provider`、`permission_policy`、`system_prompt`、`tool_allowlist`、`network_scope`、`filesystem_scope`、`runtime_budget`、`max_parallel_subagents` 等信息。

# Skill 标准
digagent 原生采用开放的 Agent Skills / `SKILL.md` 生态作为外部 skill 标准。支持渐近加载，先看 metadata，需要时再读 `SKILL.md`，再按需读 `references/`、`scripts/`。

## 目录规范
每个 skill 为一个目录，至少包含：
- `SKILL.md`

可选包含：
- `scripts/`
- `references/`
- `assets/`
- `agents/openai.yaml`

## SKILL.md 规范
必须使用 YAML frontmatter，至少包含：
- `name`
- `description`

建议增加：
- `version`
- `entrypoints`
- `inputs`
- `recommended_tools`
- `risk_level`

## 兼容原则
- digagent 兼容 Codex / Claude Code 所采用的 `SKILL.md` skill 生态。
- digagent 内部将外部 skill 统一转换为本地 `SkillManifest`。
- skill 不直接拥有执行权限。
- skill 的脚本、文件访问、网络访问、工具调用均必须经过 digagent permission layer。

# 核心数据模型
## Session
Session 是用户与系统的长期会话容器，一个 session 下可以有多个 run。

推荐文件：
- `data/sessions/{session_id}/session.json`
- `data/sessions/{session_id}/messages.ndjson`
- `data/sessions/{session_id}/runs/{run_id}.json`

`session.json` 建议结构：

```json
{
  "schema_version": "1.0",
  "session_id": "sess_xxx",
  "title": "分析某 Web 靶场题",
  "created_at": "2026-04-08T09:50:00Z",
  "updated_at": "2026-04-08T10:20:00Z",
  "status": "idle|active_run|awaiting_approval|awaiting_user_input|archived",
  "root_agent_profile": "sisyphus-default",
  "intent_profile": {
    "objective": "分析目标站点公开面",
    "labels": ["web_analysis"],
    "report_kind_hint": "pentest_report",
    "confidence": 0.81
  },
  "scope": {
    "repo_paths": [],
    "allowed_domains": ["target.example.com"],
    "artifacts": ["challenge.zip"]
  },
  "run_ids": ["run_xxx"],
  "memory_refs": ["mem_x1"],
  "evidence_refs": ["ev_x1", "ev_x2"],
  "report_refs": ["rep_x1"],
  "last_report_id": "rep_x1"
}
```

`messages.ndjson` 每行一条消息，便于流式追加：

```json
{
  "message_id": "msg_xxx",
  "session_id": "sess_xxx",
  "run_id": "run_xxx",
  "role": "user|assistant|system|tool",
  "sender": "user|sisyphus|hephaestus_01|tool_runner",
  "content": "请分析这个附件",
  "artifact_refs": [],
  "created_at": "2026-04-08T09:51:00Z"
}
```

## Memory
Memory 采用 OpenClaw 式分层结构：长期记忆、daily notes、wiki 编译知识三层分开保存。一次性对话、工具原始输出和 run 元数据不得直接写入长期记忆。

推荐文件：
- `data/memory/MEMORY.md`
- `data/memory/daily/YYYY-MM-DD.md`
- `data/memory/wiki/{entry_id}.json`
- `data/memory/index.json`

长期 memory 推荐保存“跨 run 复用价值高、且有 evidence 支撑”的内容。daily notes 保存日记式过程摘要，wiki 保存编译后的 claim/evidence 视图。

`MEMORY.md` 示例：

```markdown
# DigAgent Memory

## operator_preference: 用户偏好 Markdown 报告并保留证据引用

{
  "preference": "markdown_with_evidence_refs"
}
```

`daily/YYYY-MM-DD.md` 示例：

```markdown
## Run run_xxx

目标：分析某 Web 靶场题
结论：已完成当前范围内的公开取证，并生成报告。

Evidence: `ev_x1`, `ev_x2`
```

`wiki/{entry_id}.json` 建议结构：

```json
{
  "entry_id": "wiki_xxx",
  "title": "目标站点公开面观察",
  "summary": "首页返回 200，标题为某高校官网",
  "source_session_id": "sess_xxx",
  "source_run_id": "run_xxx",
  "claims": [
    {
      "claim": "目标首页当前可访问，公开面观察已落地为 evidence",
      "evidence_refs": ["ev_x1"],
      "confidence": 0.88
    }
  ],
  "tags": ["web_analysis"],
  "created_at": "2026-04-08T10:05:00Z",
  "updated_at": "2026-04-08T10:05:00Z"
}
```

写入规则：
- 只有 root agent 可写长期 memory。
- memory 必须来自 evidence 支撑，不能直接把模型猜测写入。
- 高敏感 memory 默认不跨 session 自动注入。
- run 元数据、完整聊天原文和原始工具输出只能进入 daily notes，不直接进入 `MEMORY.md`。

## Evidence
Evidence 是所有结论的最小可追溯单元，工具输出、网页片段、代码片段、子 agent 结论都应先落为 evidence。

推荐文件：
- `data/evidence/{evidence_id}.json`

建议结构：

```json
{
  "evidence_id": "ev_xxx",
  "session_id": "sess_xxx",
  "run_id": "run_xxx",
  "type": "tool_output|web_content|code_snippet|subagent_result|user_input",
  "title": "反序列化入口代码片段",
  "summary": "在 app/controllers/demo.py 中发现 pickle.loads",
  "source": {
    "tool_name": "grep_code",
    "agent_id": "hephaestus_01",
    "path": "/workspace/project/app/controllers/demo.py",
    "url": null
  },
  "artifact_refs": ["art_x1"],
  "hash": "sha256:...",
  "content_ref": "artifact://art_x1",
  "structured_facts": [
    {
      "key": "dangerous_api",
      "value": "pickle.loads"
    }
  ],
  "confidence": 0.95,
  "sensitivity": "medium",
  "created_at": "2026-04-08T10:10:00Z"
}
```

使用规则：
- 报告中的每个关键结论至少引用一个 evidence。
- evidence 与 artifact 分离，evidence 保存“结构化结论”，artifact 保存“原始材料”。

## Artifact
Artifact 保存原始文件、截图、下载结果、日志全文等大对象。

推荐文件：
- `data/artifacts/{artifact_id}.json`
- `data/artifacts/blob/...`

建议结构：

```json
{
  "artifact_id": "art_xxx",
  "kind": "file|stdout|stderr|html|screenshot|download",
  "session_id": "sess_xxx",
  "run_id": "run_xxx",
  "storage_path": "data/artifacts/blob/run_xxx/stdout_01.txt",
  "mime_type": "text/plain",
  "size_bytes": 1204,
  "sha256": "...",
  "created_at": "2026-04-08T10:10:00Z"
}
```

## Report
Report 是用户可读交付物，先由专门写作 agent 生成结构化 draft，再由 deterministic validator 校验，最后统一渲染为 Markdown/PDF。

推荐文件：
- `data/reports/{report_id}.json`
- `data/reports/{report_id}.md`
- `data/reports/{report_id}.pdf`

建议结构：

```json
{
  "report_id": "rep_xxx",
  "session_id": "sess_xxx",
  "run_id": "run_xxx",
  "kind": "writeup|pentest_report|code_review_report",
  "title": "某题目分析报告",
  "scope": {
    "targets": ["challenge.zip"],
    "constraints": ["no destructive exploit"]
  },
  "summary": "成功定位并解释漏洞成因",
  "findings": [
    {
      "finding_id": "fd_x1",
      "title": "存在不安全反序列化",
      "severity": "high",
      "confidence": 0.93,
      "claim": "应用在未校验输入的情况下调用 pickle.loads",
      "evidence_refs": ["ev_xxx"],
      "reproduction_steps": [
        "打开 demo.py",
        "定位到 pickle.loads 调用点"
      ],
      "remediation": "改用安全序列化方案并增加输入校验"
    }
  ],
  "limitations": [
    "未进行真实攻击验证，仅进行静态分析"
  ],
  "writer_summary": "我已基于当前任务图、已完成节点和 evidence 写出最终报告草稿。",
  "evidence_refs": ["ev_xxx"],
  "export_paths": {
    "markdown": "data/reports/rep_xxx.md",
    "pdf": "data/reports/rep_xxx.pdf"
  },
  "generated_at": "2026-04-08T10:20:00Z"
}
```

约束：
- Markdown 是主渲染源，PDF 由 Markdown 渲染得到。
- 报告中的每个 finding 必须有 `evidence_refs`。
- 报告导出动作本身也要进入 audit。
- `kind` 由 writer agent 基于 `intent_profile + task_graph + evidence` 推断，不能再由旧 `task_type` 字段硬编码决定。

# 用户接口

WebUI 采用双视图模式，用户可随时在聊天视图与流程图视图间切换，满足对话交互与任务可观测性需求。

## 视图模式

### 1. 聊天视图（默认）
常规对话界面，自上而下显示消息流，支持：
- 用户输入框，支持多行文本。
- 消息气泡区分用户、Sisyphus、subagent、系统通知。
- 当 run 进入 `waiting_user_input` 时，聊天流追加一条普通 assistant 气泡显示 clarify 问题，文案与 task node 的 `metadata.question` 保持一致。
- 审批卡片内嵌展示：当 run 进入 `awaiting_approval` 时，在对话流中渲染一张包含动作摘要、风险标签、通过/拒绝按钮的卡片。
- 证据引用：助理消息中的结论可点击跳转至对应 evidence 详情面板。
- 工具栏提供“导出报告（Markdown/PDF）”、“查看当前 run 摘要”、“取消任务”等操作。

参考[openclaw](https://github.com/openclaw/openclaw)的webui给用户的接口

### 2. 流程图视图
以任务规划 DAG 形式展示当前 run 的执行步骤、阻塞点与完成情况，便于观察进度与任务分支。
- 节点按当前 run 的 task node / 子任务 / 汇总步骤绘制，当前激活节点高亮。
- 节点状态用颜色标识：待执行（灰）、运行中（蓝）、等待审批/输入（橙）、已完成（绿）、失败（红）。
- 点击节点可展开详情：展示节点摘要、关联 evidence、原始动作摘要；若节点处于审批等待，则可直接唤起审批卡片。
- 若 run 尚未创建，流程图视图显示“无活跃任务”。

参考[LuaN1aoAgent](https://github.com/SanMuzZzZz/LuaN1aoAgent)如何展示任务规划图

## 视图切换
界面顶部提供标签页（聊天 / 流程图）或浮动切换按钮，点击后即时切换，不丢失对话状态。

## 侧边栏信息
左侧常驻线程栏以 session 列表为主，支持搜索、新建 session、切换历史会话。
当前 run 的预算使用、历史 run、漏洞知识库同步状态等辅助信息收纳到“任务详情”抽屉，不再默认占据主版面。

## 布局与滚动
- WebUI 使用固定视口壳，而不是整页随消息增长继续拉长。
- 左侧 session 列表、聊天消息区、任务图区、节点详情区各自内部滚动。
- 输入框固定在聊天视图底部，长对话只推动消息区滚动，不推动整个页面滚动。

## 流式输出支持
聊天视图中，agent 的思考过程与工具调用结果以流式增量形式追加，避免长时间空白等待。流程图视图中的节点状态随 run 进展实时更新。

## 审批交互
审批卡片在聊天视图中展示为独立区域，在流程图视图中点击处于 `waiting_approval` 的 task node 也可直接弹出。用户批准后，UI 生成 approval token 随请求发送至后端，run 自动继续执行。

# 验收标准
## 功能验收
CLI 可启动主 agent
WebUI 可发起任务并看到流式输出
可加载至少 2 个 agent profile
可注册至少 2 个 tools 和 1 个 skill
可导出 Markdown
可导出 PDF

## 集成验收
调用程序完成一道密码学 CTF 题的解答，例如：一只小羊翻过了 2 个栅栏 `fa{fe13f590lg6d46d0d0}`，应得到 `flag{6fde4163df05d900}`
能完成 1 个源码分析任务，例如对本项目
能完成 1 个简单 Web 分析任务，例如对 `www.xjtu.edu.cn` 进行分析

## 安全验收
未授权工具调用被拒绝
高风险调用进入审批
所有工具调用有审计日志
skill 无法绕过 permission 层
