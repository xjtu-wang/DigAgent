# DigAgent 答辩备份资料

## 1. 核心类型速查

### 1.1 Session 与 Run

| 类型 | 作用 | 关键字段 |
|---|---|---|
| `SessionRecord` | 长期对话容器 | `session_id, status, scope, run_ids, active_run_id` |
| `RunRecord` | 单次受控执行 | `run_id, status, user_task, scope, task_graph, evidence_ids, approval_ids, report_id` |

### 1.2 TaskGraph

| 类型 | 作用 | 关键字段 |
|---|---|---|
| `TaskGraph` | DAG 真相 | `run_id, nodes, edges, graph_version` |
| `TaskNode` | 执行节点 | `node_id, kind, status, description, summary, owner_profile_name, metadata` |
| `TaskEdge` | 依赖关系 | `source, target` |
| `GraphEditOp` | 局部重规划补丁 | `op_type, node, edge, patch` |

`TaskNodeKind`：`input / tool / skill / subagent / aggregate / report / export`

### 1.3 权限与审批

| 类型 | 作用 | 关键字段 |
|---|---|---|
| `ActionRequest` | 统一动作提议 | `action_type, name, arguments, targets, risk_tags` |
| `PermissionOutcome` | 权限决策 | `decision, reason, normalized_targets` |
| `ApprovalRecord` | 审批真相 | `approval_id, status, action_digest, policy_key, challenge` |
| `DelegationGrant` | subagent 衰减授权 | `delegatee_profile_name, allowed_tools, allowed_paths, allowed_domains, max_tool_calls` |

### 1.4 证据与报告

| 类型 | 作用 | 关键字段 |
|---|---|---|
| `ArtifactRecord` | 原始产物 | `artifact_id, kind, storage_path, mime_type` |
| `EvidenceRecord` | 可追溯证据 | `evidence_id, title, summary, source, artifact_refs` |
| `ReportDossier` | 写报告输入包 | `intent_profile, task_graph, evidence, artifacts, retrieved_memory` |
| `ReportDraft` | writer 输出草稿 | `kind, summary, findings, limitations, evidence_refs` |

## 2. Profile 矩阵

| Profile | 定位 | tool_allowlist 特征 | 典型产物 |
|---|---|---|---|

## 4. Skill / Plugin / MCP 对比

| 维度 | Skill | Plugin | MCP |
|---|---|---|---|
| 本质 | 知识层 | 本地能力包 | 外部 server 能力 |
| 载体 | `SKILL.md` + references | `.plugin/plugin.json` + commands | `config/mcp/servers/*.yaml` |
| 是否直接执行 | 否 | 是 | 是 |
| 是否受 permission 约束 | 经 `skill_consult` 间接受控 | 是 | 是 |
| 当前默认是否可执行 | 可加载 | 可执行 | 默认 `enabled: false` |

## 8. 高频 Challenge 问答

### Q1：是 prompt 系统还是 runtime 系统？

A：两者都有，技术核心在 runtime。Prompt 只定义角色与输出契约，执行控制在 `SessionManager + GraphManager + PermissionEngine + FileStorage`。

### Q2：为什么不让一个大模型把规划、执行、写报告全做了？

A：不同阶段的正确性标准不同。规划要结构化图、执行要严格权限、报告要 evidence-backed、记忆要稳定复用。混在一个角色里边界必然崩。

### Q3：为什么说是 multi-agent，但不允许 subagent 继续委派？

A：这是受控 multi-agent。目标不是自治层级扩张，而是切专项，同时保留 root supervisor 的全局控制权。

### Q4：为什么 skill 不能直接执行？

A：skill 属于知识层。让 skill 承载执行权限会把"知道怎么做"和"有权做什么"混在一起，安全边界变差。

### Q5：MCP 接入了吗？

A：client、manifest、catalog、permission、stdio 通路都已接入，但 `playwright-local` 与 `kali-local` 默认 `enabled: false`。架构已接好，执行面保守关闭。

### Q6：memory 是 RAG 吗？

A：不是流行意义上的向量 RAG。当前是文件型分层记忆 + BM25 scoped retrieval，优先可审计和 evidence 绑定。

### Q7：老师说"你们只是把 LangChain 套一层壳"怎么答？

A：项目引入了 LangChain 生态依赖，但主执行路径是自研 harness，session / run / task graph / approval / audit 这条链完全自研。

### Q8：报告为什么可信？

A：report 不从聊天文本生成，而是基于 dossier，validator 强制每条 finding 引用合法 `evidence_refs`。

### Q9：审批为什么要区分 digest 和 policy key？

A：动作完全相同与策略等价是两个问题。digest 服务审计，policy_key 服务审批复用与 supersede。

### Q10：最大 gap 是什么？

A：**并行 ready 节点端到端耗时约 1.50s，未达阈值 0.55s**；此外 pyproject 声明了 LangChain 生态但主执行是自研 runtime；MCP 默认 enabled 为 false。三条 gap 都会主动讲。

## 9. 术语统一口径

- Prompt：角色与输出契约
- Context：阶段化最小必要事实注入
- Harness：运行时控制骨架
- Tool：统一动作入口
- Skill：知识 / 方法论 bundle
- Plugin：本地能力 bundle
- MCP：可选外部 server 能力
- Artifact：原始产物
- Evidence：可引用证据
- Dossier：报告输入包
- Curator：记忆整理代理

## 10. 一句话备用总结

如果老师要求一句话总结：

> DigAgent 把安全分析任务做成了一个有 prompt 分层、有上下文组装、有 DAG 执行、有权限审批、有证据与报告闭环的受控运行时——创新点不是让模型多调几个工具，而是把每一步都锁进工程边界里。
