# DigAgent 技术报告

## 1. 项目定位与答辩口径

**本章回答：DigAgent 是什么，以及用什么口径讨论它。**

DigAgent 是一个受控安全分析 agent，不是通用聊天机器人。核心目标不是把模型包装成会说话的工具，而是把一次安全分析任务做成可审计、可恢复、可导出、可审批的运行过程。项目有两个入口：

- CLI：`digagent run/chat/approve/serve/cve-sync/cve-status/cve-search/tools-doctor`
- WebUI：React + Vite 前端，通过 FastAPI 提供 REST 与 SSE

本报告遵循"现状实现 + 设计思想 + 已知差距"三段式口径：

- 已实现事实只来自当前仓库的代码、配置、接口、测试与 2026-04-18 本地验证结果
- 设计思想可以引用 `docs/项目设计.md`，但必须与现状明确区分
- 已知差距主动暴露，不以模糊表述回避老师追问

## 2. 外部范式参考与本项目映射

**本章回答：我们借鉴了谁，落地到什么程度。**

### 2.1 Claude Code 范式

项目借鉴 Claude Code 的三层拆分：`prompt / context / harness`。

| 参考思想 | DigAgent 对应实现 | 当前边界 |
|---|---|---|
| Prompt 不是单一长 system prompt，按角色和任务拆分 | `config/agents/*.yaml` + `config/prompts/**` + `src/digagent/prompts.py` | 已实现 |
| Context 按当前阶段组装最小必要事实 | planner / scheduler / direct-answer / subagent 各自注入不同上下文 | 已实现 |
| Harness 控制状态、工具、权限、审批、审计 | `SessionManager / GraphManager / ToolRegistry / PermissionEngine / FileStorage` | 已实现 |

### 2.2 oh-my-opencode 的 multi-agent 定位

项目借鉴"root supervisor + specialist"定位，而非平级自治：

- `sisyphus-default`：root supervisor
- `prometheus-planner`：规划、追问、批次选择
- `hephaestus-deepworker`：仓库与网络深挖
- `hackey-ctf`：CTF 专项执行
- `report-writer`：报告草稿
- `memory-curator`：分层记忆整理

关键点：subagent 不是独立主脑，而是 root 在受限授权下放出的执行切片。它拿不到长期记忆写权限，也不能继续委派下一层 subagent。

### 2.3 OpenClaw 的 memory 思路

项目借鉴"自进化分层记忆"，不把所有历史塞进向量库。当前实现全部落在文件系统：

- `MEMORY.md`：长期摘要
- `memory/*.json`：结构化长期记忆条目
- `memory/daily/YYYY-MM-DD.md`：日报型运行记录
- `memory/wiki/*.json`：可复用 claim 视图
- `memory/index.json`：索引
- `MemorySearchEngine`：基于 BM25 的 scoped retrieval

因此 DigAgent 的 memory 是文件型、可审计、带 evidence 引用的记忆系统，不是向量数据库平台。

## 3. 总体架构

**本章回答：五层是哪五层，以及各自的职责边界。**

```
Prompt       角色与输出契约
Context      阶段化最小必要事实
Harness      状态机 / 图执行 / 权限 / 审批 / 审计
Capability   tool / skill / plugin / MCP
Persistence  文件落盘 + REST + SSE + WebUI
```

五层的分工原则是"可替换 + 不相互污染"：换 prompt 不影响运行时，换执行骨架不影响 tool 目录，换前端不影响持久化。每一层的具体实现放在 §4–§6 讨论。

## 4. Prompt 层

**本章回答：prompt 怎么组装、怎么约束、怎么拆角色。**

### 4.1 Prompt 作为磁盘资源

`load_prompt_text()` 与 `load_runtime_prompt()` 把 prompt 当成磁盘资源读取。收益有三点：prompt 可按角色拆分维护、共享规约集中在 `_shared`、切换 profile 不改运行时代码。planner 的 system prompt 最终不是字符串常量，而是共享语言约束、共享 contract、角色说明等片段的拼装结果。

### 4.2 先约束行为，再约束格式

Prompt 层包含两类约束：

1. **行为约束**：输入视为不可信数据（prompt / HTML / JSON / 日志 / 注释），先被动观察再主动探测；skill 只加载上下文，不承载执行；证据不足写 limitations，不编造。
2. **输出约束**：`AgentBridge.JSON_OUTPUT_DISCIPLINE` 强制只输出原始 JSON；planner / scheduler / router / writer / curator 各有明确 schema；不合法 JSON 触发 retry，仍失败进入 repair。

### 4.3 角色分离，不让一个模型既当裁判又当运动员

| Profile | 角色 | 典型输出 |
|---|---|---|
| `sisyphus-default` | root supervisor | 用户可见答复、生命周期控制 |
| `prometheus-planner` | planner / scheduler / router | PlanningBundle、message route、execution batch、graph ops |
| `hephaestus-deepworker` | repo / web 专项 worker | 结构化子任务结果 |
| `hackey-ctf` | CTF 专项 worker | 题目执行结果 |
| `report-writer` | 报告 writer | `ReportDraft` |
| `memory-curator` | 记忆整理 | daily_note / memory_candidates / wiki_entries |

不同阶段正确性标准不同：规划要结构化图，执行要严格权限，报告要 evidence-backed，记忆要稳定可复用。混在一个 prompt 里只会让边界模糊。

### 4.4 生产路径不允许 silent fallback

`AgentBridge` 对 planner / writer / router / scheduler 的处理统一 fail-fast：无模型配置直接报错、JSON 为空直接报错、schema 不匹配显式 retry/repair、不能解析显式抛异常。只有在 `DIGAGENT_USE_FAKE_MODEL=1` 测试模式下才允许测试桩。这一点证明 Debug-First 原则真的落到了实现里。

## 5. Context 层

**本章回答：不同阶段的模型各自看到什么。**

### 5.1 Planner：做决策所需的全部能力面

`plan_task_graph()` 注入的核心字段：`run_id / task / scope / followup_messages / tool_allowlist / available_tools / available_specialists / available_plugins / available_skills / available_mcp_servers / memory_hits`。相当于告诉 planner：任务是什么、用户授权什么范围、能调什么、能委派给谁、过去记忆里有哪些稳定知识。

### 5.2 Direct answer：当前状态快照

用户问"为什么要审批""当前进度是什么"时，系统走 `compose_direct_answer()` 而非重跑 planner。注入字段包括 `session_status / run_status / graph / 最近 evidence 摘要 / memory hits / pending approvals / awaiting reason / budget 使用情况`。这保证过程性问答是在解释当前状态，而不是偷偷继续执行。

### 5.3 Subagent：衰减后的 grant

subagent worker prompt 注入 `grant.allowed_tools / allowed_paths / allowed_domains`、`task.goal` 与 `task.evidence_summaries`、worker 历史、MCP server catalog、`enabled_server_ids`、skill/plugin 到 `server_id` 的 alias hints。目的有二：避免 subagent 拿到过大的全局上下文；避免 subagent 把 skill 名或 plugin id 误当成 MCP `server_id`。

### 5.4 Skill 是 context，不是 capability

`SkillCatalog` 读取 `data/skills/*/SKILL.md`，skill 可带 `references/`、声明 `recommended_tools`、带 `agents/openai.yaml` 接口说明。`allow_implicit_invocation` 只影响调度提示，不授予执行权限。skill 的作用是"给模型加载方法论"，不是直接执行 shell、网络、MCP 或 patch。

## 6. Harness 层

**本章回答：什么在控制系统状态。**

### 6.1 Session 是长期容器，Run 是单次受控执行

系统最顶层不是消息列表，而是 `SessionRecord + RunRecord`：

- `SessionRecord` 持有消息、scope、run 列表、active run
- `RunRecord` 持有当前任务的 DAG、证据、审批、报告、记忆候选

这让系统能分别回答"当前 session 是否允许新建 run"和"当前 run 执行到哪一步"。

### 6.2 SessionRouter 决定消息去向

`handle_message()` 把用户消息分成 `direct_answer / clarification_input / approval_response / cancel / new_run_request` 五类，对应 `create_run / continue_run / reject / direct_answer` 四种处置。同一 Session 同一时刻只允许一个 active Run，是避免多 run 并发失控的关键约束。

### 6.3 Run 是 TaskGraph，不是步骤列表

`TaskNodeKind` 包含 `input / tool / skill / subagent / aggregate / report / export`。`GraphManager.refresh()` 负责：重算 `depends_on / children`，维护 `ready / active / completed / blocked / deprecated` 节点集，校验 DAG 无环。这让系统能自然表达：clarify 节点阻塞后续路径、两个独立 tool 节点并行 ready、report/export 只能在 aggregate 之后。

### 6.4 Scheduler 与分发是两层结构

执行不是"拿 ready 节点直接跑"：

1. planner 先用 `select_execution_batch()` 选出本轮批次
2. runtime 再用 `_dispatch_ready_nodes()` 实际分发

runtime 区分纯内部节点（直接执行）与外部节点（tool 或 subagent，需先走 `_prepare_external_node()`）。外部节点先经过权限决策，再进入并发 `asyncio.gather()`。

### 6.5 Clarify 后 supersede 旧分支

用户补充信息后，`_replan_after_user_input()` 重新拿到 planning bundle，再由 `_replace_clarified_branch()` 把 clarify 之后的旧分支节点标为 `deprecated`、记录 `superseded_by`、把新分支接在 clarify 节点之后、递增 `planning_phase`、为新节点记录 `replanned_from_node_id`。这证明澄清是图级别的分支替换，不是文本级别的追加。

### 6.6 Budget 与 timeout 是内建约束

运行时维护 `tool_calls_used / runtime_seconds_used / active_subagents / active_tools`。profile 声明 `max_parallel_tools / max_parallel_subagents / max_tool_calls / max_runtime_seconds`。超预算触发 deny 或 timeout，不静默降级。

## 8. Tool / Skill / Plugin / MCP 的分层设计

**本章回答：四类能力为什么要拆开，怎么统一纳管。**

### 8.1 Tool：manifest + adapter

`ToolRegistry` 从 `data/tools/*.yaml|json` 读 manifest，按 `executor_adapter` 分发到 `WorkspaceToolset / NetworkToolset / McpToolset / plugin command`。manifest 关键字段：`name / description / risk_tags / default_targets / executor_adapter`。工具目录是配置数据，不是写死的 if/else。

### 8.2 Skill：知识层

以 `ctf-sandbox-orchestrator` 为例：入口 skill、声明 competition 家族路由规则、带 `references/web-api.md` 与 `references/reverse-native.md` 等参考索引、`policy.allow_implicit_invocation: true`。skill 本身不执行任何 shell 或 MCP 操作。

### 8.3 Plugin：本地能力包

仓库已有 `ctf-sandbox-orchestrator` plugin，包含 `.plugin/plugin.json`、`commands/*.yaml`、`scripts/*.py`。plugin command 动态合并进 `ToolRegistry.catalog()`，因此 plugin 能力与内置工具走同一套 permission / audit 链。

### 8.4 MCP：stdio-first，默认保守

`config/mcp/servers/*.yaml` 当前典型样例：`playwright-local` 与 `kali-local`，transport 均为 `stdio`，默认 `enabled: false`。catalog 可发现，但 disabled 状态下不能静默调用。这是安全边界设计，不是功能未完成。

## 9. Permission、Approval、Audit

**本章回答：高风险动作怎么被拦住、怎么被审计。**

### 9.1 权限决策三段式

`PermissionEngine.decide()` 输出 `ALLOW / CONFIRM / DENY` 三态。决策依据：action 是否注册、是否在 profile `tool_allowlist` 内、path 是否落在 `filesystem_scope` 与 session scope 内、domain 是否落在 `network_scope` 内、MCP server 是否在 `mcp_server_allowlist` 内、runtime budget 是否超限、risk tags 是否命中高风险集合。

### 9.2 命中高风险标签进入 CONFIRM

高风险标签集合：`filesystem_write / shell_exec / network / external_exploit / export_sensitive`。`web_fetch`、高风险 MCP 调用、PDF 导出等都可能触发审批。

### 9.3 审批是带摘要的 challenge-token 机制

系统区分两种摘要：

- `approval_digest`：审计级动作指纹，严格对动作身份做哈希
- `policy_key`：策略等价键，忽略 run / node / actor 等身份字段，只看"做了什么"

作用不同：digest 判断当前批准的动作是不是原动作；policy_key 判断两个动作是否策略等价、可否沿用既有审批。动作变了但策略仍等价，可以更新 digest；策略键也变了，就 supersede 旧审批生成新审批。

### 9.4 审计链贯穿全程

所有动作都记录 audit event，典型 executor 包括 `policy / tool_runner / subagent_runner / report_runner / skill_runner`。可以回答"你们怎么证明系统真的经过审批与审计，而不是 prompt 里说说而已"——执行器名称直接写在事件里。

## 10. Multi-agent 设计

**本章回答：root 和 subagent 的权力边界在哪里。**

### 10.1 Root supervisor 持有全局控制权

`sisyphus-default` 负责 session 路由、run 生命周期、审批协调、最终 assistant message、report/export、memory commit。

### 10.2 Planner 输出结构化图

`prometheus-planner` 输出 `PlanningBundle`：`intent_profile / planner_message / clarify_message / task_graph`。比 prose plan 强的地方是 runtime 能直接验证每个节点的 kind、tool_name、owner_profile_name、arguments 是否合约束。

### 10.3 Subagent 是受限 worker

subagent 节点执行时生成 `DelegationGrant`，含 `delegatee_profile_name / allowed_tools / allowed_paths / allowed_domains / max_tool_calls`。`allowed_tools` 取 root profile 与 worker profile allowlist 的交集，并明确排除 `delegate_subagent / report_export / skill_consult`。subagent 只能做专项执行，不能无限套娃，也不能越权碰报告和记忆。

### 10.4 Worker 输出也走 schema

worker 允许的输出只有两种：`{"type":"tool_call", ...}` 与 `{"type":"final", ...}`。其他输出 runtime 直接报错。subagent 的"思考"被约束成有类型的动作提议。

## 11. Evidence、Report、Export

**本章回答：报告凭什么可信。**

### 11.1 Artifact 是原始产物，Evidence 是可引用证据

tool 或 subagent 结果先保存为 artifact（原始 stdout / json / html / pdf / 截图），再抽象成 evidence（带 `title / summary / source / structured_facts / artifact_refs`）。artifact 保留原始信息便于复查，evidence 便于 planner / writer / UI / memory 直接消费。

### 11.2 Report 先写 draft 再校验

流程：

1. `_build_report_dossier()` 汇总 task graph、evidence、artifacts、memory hits
2. `report-writer` 生成 `ReportDraft`
3. `ReportValidator` 校验 kind 是否与任务意图一致、finding 是否有合法 `evidence_refs`、summary 是否为空或与任务目标冲突
4. `ReportExporter` 输出 Markdown，再导出 PDF

### 11.3 PDF 导出显式依赖

优先调用 `webui/render-pdf.mjs`，否则退到 headless Chrome CLI。无论走哪条路径：renderer 异常显式失败，没生成输出文件也显式失败。与 Debug-First 一致。

## 12. Memory 设计

**本章回答：谁能写记忆、记什么、怎么取。**

### 12.1 只有 root agent 可提交长期记忆

`_commit_memory()` 只在 run 完成后由 root runtime 触发。subagent 结果可进入 evidence，但不能直接写长期 memory。

### 12.2 Curator 做"提升"，不是总结全文

`memory-curator` 输出固定为 `daily_note / memory_candidates / wiki_entries`：daily note 记录本次 run 日报摘要；memory candidate 只收敛"可跨 run 复用"的稳定知识；wiki entry 强调 claim 和 `evidence_refs`。

### 12.3 检索是 scoped BM25

`MemorySearchEngine` 从 `MEMORY.md`、memory records、wiki、daily docs 构造文档集，按 `scope=session|run|global` 过滤，按 sensitivity 过滤，BM25 打分，对同 session / 同 run 结果额外加权。memory 更像"带范围控制的工程知识检索"，而非"大而全回忆库"。

## 13. API、CLI、WebUI 实现

**本章回答：系统怎么被外部使用。（完整路由清单见答辩备份资料 §5）**

### 13.1 API 分组

FastAPI 路由分为六组：`health / catalog / settings`、`session`、`run`、`approval`、`evidence + artifact + report`、`cve`。以 session 组为例，覆盖创建、列表、详情、scope patch、消息、runs、archive、unarchive、delete、events 全生命周期。

### 13.2 CLI 命令

共 8 个：`digagent run / chat / approve / serve / cve-sync / cve-status / cve-search / tools-doctor`。覆盖一次性执行、交互式会话、审批确认、后端启动、CVE 同步/搜索、工具体检。

### 13.3 WebUI 关键能力

不是"做个聊天框"，而是把 runtime 状态可视化：会话侧边栏、聊天主时间线、关键系统卡片、任务图画布、运行活动流、evidence inline 展开、report inline 查看与下载、approval 卡片批准/拒绝、浏览器端本地设置存储。前端通过 `EventSource` 订阅 session 级 SSE，把 run 事件实时合并到前端时间线和图谱。

## 14. 测试覆盖与可证明性

**本章回答：系统被测到什么程度，哪里没测达标。**

已覆盖的关键场景：CTF 题到 writeup、源码分析到 code review report、Web 分析触发审批、clarify 后继续 run、clarify 阶段 direct answer、单条消息多意图仍保持单 run、archived session 仍可问答、profile 驱动全角色分工、skill 节点只加载上下文、planner JSON parsing / repair、审批 digest / policy key 稳定性、timeout 与 approval expiry、plugin command 入图执行、layered memory 落盘、MCP allowlist 安全约束。

**2026-04-18 对核心测试集做只读验证，结果 37 通过 / 1 失败；失败项 `tests/test_scheduler_and_manifest.py::test_non_exclusive_ready_nodes_run_in_parallel`，并行 ready 节点端到端耗时约 1.50s，未达到阈值 <0.55s。** 答辩时的准确表述是"系统支持并行 ready 节点调度语义，但并发性能尚未达到该测试设定的阈值"，不能说成"并发能力已完全达标"。

## 15. 已知差距

**本章回答：主动暴露的三条事实。**

### 15.1 并行调度性能未达测试阈值

调度语义已支持并行启动，性能优化待跟进。这是当前最明确的、已被测试暴露的 gap。

### 15.2 依赖声明与真实主路径存在差异

`pyproject.toml` 声明了 `langchain / langgraph / deepagents`，但主执行路径是自研 `SessionManager + GraphManager`，仅在模型桥接处使用 `langchain_openai.ChatOpenAI`。准确表述："项目引入了 LangChain 生态依赖，但主执行 harness 主要是自研 runtime"，不能说成"我们完全基于 LangGraph 执行图跑起来了"。

### 15.3 MCP 默认保守接入

`playwright-local` 与 `kali-local` 的 `enabled` 默认为 `false`，catalog 可见但不可静默执行。这是安全边界设计，不是功能未完成。

## 16. 结论

DigAgent 的价值不在于"能调用很多工具"，而在于把安全分析过程做成了受控运行时。三条核心主张：

1. **工程边界清晰**：prompt 负责角色契约、context 负责阶段化事实、harness 负责状态与权限、capability 分层纳管、证据与报告闭环——每一层都可替换、可审计。
2. **执行真相文件化**：session / run / evidence / approval / report 全部落盘，模型上下文不是真相来源。
3. **主动暴露差距**：区分设计目标和实现现状，把并行性能、依赖声明、MCP 默认关闭三件事讲透，而不是藏着。
